# 写单元测试的顺序

## 写单测的顺序一般有三种

1. `写测试、自动化验证调试` ==> `写功能` 👍👍👍

   TDD，是最推荐的方式。

2. `写功能` ==> `写测试、自动化验证调试` 👍

   用自动化验证调试，代替了手动去验证调试。

3. `写功能` ==> `手动验证/调试` ==> `写测试、自动化验证调试` 👎

   最痛苦的，会让人对测试感到厌恶。

## TDD 三部曲

![](assets/test-driven-development-TDD.webp)

1. **写一个测试（还没法通过）**

   这一步能让我们梳理清楚需求，写测试的同时，我们也在设计程序的对外接口。

2. **写一个让测试通过的业务代码**

   根据测试的报错，来一步一步写能让测试通过的代码。

   > 我们根据测试的报错，来解决一个又一个的问题。
   > TDD 的过程就像是一个闯关游戏。

3. **重构**

> **TDD 是敏捷开发的核心要素。**
>
> 没有 TDD 的敏捷开发，都是假敏捷。
